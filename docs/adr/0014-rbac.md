# ADR-0014: Role-based access control (RBAC)

- Status: accepted (2026-06-27)
- Phase: post-roadmap (frontend redesign + RBAC)
- Related: ADR-0007 (auth: JWT + refresh), ADR-0004 (PG truth / Redis
  rebuildable), ADR-0013 (security posture / threat model)

## Context

Until now every authenticated user had identical privileges. The Phase-8
frontend redesign shipped a complete **RBAC seam** (`frontend/src/lib/rbac.ts`,
a `role` field on the user schema, `usePermissions`/`<Can>`/`AuthGate`, a gated
`/admin` route) that was deliberately dormant: the backend never sent a role, so
the schema defaulted everyone to `user` and every gate was a no-op. This ADR
records making RBAC real and **enforced server-side**, plus the first two
capabilities behind it (admin user-management and a moderator "view all
submissions" view). Content management (contests/problems) is intentionally left
to the seed for now; the permission constants for it already exist.

## Decision

**Three roles, ranked:** `user` < `moderator` < `admin`. Permissions are named
capabilities (`contest.create`, `contest.edit`, `problem.manage`,
`submission.viewAll`, `user.manage`, `admin.access`) granted per role. The
gateway references *permissions*, never raw roles, at the route boundary.

**Postgres is the source of truth; the JWT caches the role.** `users.role` is a
new `NOT NULL DEFAULT 'user'` column with a `CHECK` constraint matching the role
set (migration `00002_add_role.sql`). The role is copied into the access-token
claims at mint time, so authorization is a cheap in-memory check on every
request ‚Äî no DB read on the hot path, consistent with the "PG is truth, derived
state is rebuildable" invariant (ADR-0004). The access token is the cache; the
DB is the truth.

**Propagation window = one access-token TTL (15 min).** A role change writes the
DB immediately, but a user's *current* access token keeps its old role until it
expires. The refresh flow already re-reads the user from the DB on every
rotation, so it re-mints with the fresh role ‚Äî that is the propagation path.
`/me` and the admin user list read the role live from the DB. This staleness is
the standard JWT trade-off; for this app it is acceptable and documented rather
than engineered away (no per-request DB role lookup, no token denylist).

**The permission model is mirrored backend‚Üîfrontend.** `internal/auth/rbac.go`
and `frontend/src/lib/rbac.ts` carry the same role names, permission strings,
and rank. **The API is the only real boundary**; the frontend gating
(`<Can>`, `AuthGate`, hidden nav links) is a UX convenience. Every `/api/v1/admin/*`
route is `requireAuth` + `requirePermission(perm)` and returns 403 regardless of
what the UI shows. Unknown/invalid role strings collapse to `user` (least
privilege) on both sides.

**Endpoints added** (all enforced server-side):

- `GET /api/v1/admin/users` (`user.manage`) ‚Äî list accounts with roles.
- `PATCH /api/v1/admin/users/{id}/role` (`user.manage`) ‚Äî change a role.
  Guardrail: a user **cannot change their own role** (anti-lockout / prevents a
  sole admin self-demoting) ‚Üí 409; invalid role ‚Üí 400.
- `GET /api/v1/admin/contests/{id}/submissions` (`submission.viewAll`,
  moderator+) ‚Äî every submission in a contest, joined with the author's
  username.

## Alternatives rejected

- **DB role lookup on every request** (no role in JWT): always-fresh, but adds a
  read to the hot path and discards the stateless-token property. The 15-minute
  window does not justify it.
- **Roles inherit up the rank implicitly** (admin auto-gets every lower
  permission): convenient, but listing each role's grants in full keeps them
  explicit and auditable, and lets a higher role intentionally *lack* a
  lower-role capability later.
- **A separate `roles`/`user_roles` join table**: warranted only for many-to-many
  or per-contest roles; a single column is simpler and sufficient at this scope.

## Consequences

- Authorization is now a real, tested boundary: `auth.Can`/`AtLeast` and the
  `requirePermission` middleware have unit tests, and integration tests cover
  admin role changes (persisted + reflected after refresh), 403 for non-admins
  on every admin route, the moderator all-submissions view, and the
  self-role-change/invalid-role guardrails.
- The seed provisions one account per tier (`alice`=admin, `carol`=moderator,
  `bob`=user) so every path is exercisable by hand.
- Demoting a malicious admin is not instant ‚Äî their existing access token works
  until it expires (‚â§15 min). Revoking their refresh-token family (ADR-0007)
  ends the session at the next refresh. Recorded as an accepted residual risk,
  alongside future work: instant revocation, and the dormant `contest.*` /
  `problem.manage` permissions becoming real content-management endpoints.
